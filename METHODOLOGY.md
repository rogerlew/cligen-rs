# Methodology: How This Port Was Built, and Why You Do Not Have to Trust It

On 2026-07-09, this repository went from `git init` to a Rust
implementation of CLIGEN 5.32.3 whose binary reproduces the reference
Fortran's output **byte-for-byte** — in roughly nine and a half hours of
wall-clock time, with essentially all code written by two AI systems.

That sentence should make a careful reader skeptical. AI-generated code
fails in characteristic ways: it is plausible rather than correct, it is
confident about untested corners, it drifts from specifications no one
wrote down, and it accumulates volume faster than anyone reviews it. A
scientific model port — where a silently wrong constant produces
publishable-looking wrong numbers for years — is close to the worst
possible application for those failure modes.

This document records how the port was actually built, what its
acceptance criteria were, where its process caught real errors, and —
most importantly — how to verify every load-bearing claim yourself,
without trusting the authors, human or otherwise. The repository was
designed on the assumption that AI-generated code would be wrong in
exactly the characteristic ways; the interesting result is not that the
code was written quickly, but that the verification made the speed safe.

## 1. The acceptance criterion does not know who wrote the code

The port's correctness posture
([ADR-0001](docs/decisions/0001-source-code-authority-port.md)) is a
**source-code-authority port**: the pinned, public-domain Fortran at
[`reference/cligen532/`](reference/cligen532/PROVENANCE.md) *is* the
specification, and acceptance is **bit-identity** against a
provenance-pinned build of it — compiler version, floating-point flags
(`-O0 -ffp-contract=off -fprotect-parens -fno-fast-math`), and linked
`libm` all recorded with hashes, and the build proven deterministic by
two independent build/run cycles producing byte-identical output.

Bit-identity is a categorical criterion. A transcription error does not
score poorly against it; it fails at a specific record, and the tests
report which one. No property of the code's author — human diligence,
AI fluency — can substitute for the bits matching. This is the central
design decision: the trust question was moved from *authorship* to
*evidence*, before any code existed.

## 2. The verification ladder

Every rung below is committed, hashed, and re-runnable.

**Golden outputs.** Twelve `.cli` files generated by the pinned
reference build across four real stations (Idaho 44.97°N, California
34.23°N, Australia 30.58°S, Utah 39.83°N — both hemispheres, since the
storm-intensity model is latitude-responsive), two seed treatments, and
three modes (multi-decade stochastic generation, observed with a
sentinel-padded record, observed with a hard-truncated record, plus
single-storm).

**Interior taps.** The reference binary was instrumented — by an
additive patch to a *copy* (the vendored source is never modified;
patches and hashes committed) — to emit interior values as raw IEEE-754
bit patterns, so no decimal formatting can contaminate a comparison.
The instrumented binary's own outputs remained byte-identical to the
goldens, proving the taps non-invasive. The captures anchor every
ported unit:

| Surface | Records verified bit-identical |
|---|---:|
| `randn` uniform stream (entry-state → value vectors) | 19,784,955 |
| `dstn1` normal deviates | 26,402,148 |
| `dstg` gamma deviates (stateful replay, per-call state assertions) | 30,268 |
| `ranset` seed/QC battery (24-line stateful records) | 2,584 calls |
| ACM special functions + calendar (direct vectors) | 919 |
| Par intake + interpolator evaluators (item 4) | 692,777 |
| Daily-core day-loop replay (items 5–7, per fixture day) | 189,207 days |

The stochastic chain's quality-control coupling is included: CLIGEN 5
regenerates random batches that fail statistical tests, consuming extra
draws — a port that skipped the QC machinery would diverge silently.
The replays verify those regeneration draws draw-for-draw.

**End-to-end.** The finished binary, driven by its schema-versioned
`inp.yaml` runspec, reproduces all twelve goldens byte-for-byte
(`crates/cligen/tests/runspec_cli.rs`), including the Fortran's header
echo, its FORMAT-rounding behavior, and its diagnostic warnings at the
same simulation years.

## 3. The process caught real errors — that is the evidence it works

A verification regime is only credible if it has visibly rejected
things. The work-package record
([`docs/work-packages/`](docs/work-packages/README.md)) shows, among
others:

- **Two High-severity transcription errors** in the generator core,
  caught by cross-review against the source: single-storm mode passes
  *ordinal* day counts (e.g. 166), not calendar lengths, and two units
  had assumed 365/366 (`20260709-rng-deviates-port/artifacts/review-codex.md`).
- **Six transcendental functions rejected and re-pinned.** The
  convenient Rust math library diverged from the reference runtime at
  the 1-ULP level on millions of captured inputs (`logf` on 7.5% of
  26.4M records). Divergences of one unit in the last place bifurcate
  stochastic trajectories at branch points, so `logf`, `cosf`, `exp`,
  `tanf`, `acosf`, and `expf` were replaced with transcriptions of the
  reference platform's published algorithms (Arm optimized-routines;
  SunPro/fdlibm), each adjudicated empirically against the full capture
  and license-cleared with per-file provenance.
- **A census arithmetic error by one executor caught by the other**
  (388 vs. 381 double-precision sites), **13 findings** against the
  interface specification draft, per-package review findings with
  recorded dispositions throughout, and a characterized-and-rejected
  Fortran out-of-bounds behavior (`ranset` with `mox = 0`) that the
  port deliberately fails closed rather than replicating
  memory-layout accidents.

Errors occurred at roughly the rate one would expect. The record shows
them being caught by machinery that was in place *before* they were
made: golden fixtures before any port code, quality gates (including a
complexity-and-coverage ceiling, CRAP ≤ 30, currently 217 production
functions, none over) from the repository's first function, and
adversarial cross-review between two different AI systems on every
package — each reviewing the other's work against the Fortran source,
not against each other's opinions.

## 4. The execution model and the human layer

Every port package ran the same four-stage pipeline, with the two AI
systems in fixed, deliberately asymmetric roles:

- **Stage S — the spine (Anthropic Claude Fable 5).** The
  design-setting slice of each package, authored first: the
  instrumentation patch and its captured fixtures, the state-struct
  patterns translating the Fortran common blocks, the first ported
  units with their bit-identity tests already passing, and the
  specification work. The spine exists because whoever writes a
  package's first files is effectively writing its real coding
  standard — those files were pattern-normative for everything that
  followed. Stage S closed by producing a handoff document and the
  next stage's dispatch prompt, written *after* the code so it cites
  landed patterns rather than intentions.
- **Stage C — completion and gates (OpenAI GPT-5.6 / Codex).** The
  mechanical volume: the remaining units transcribed within the
  spine's patterns, the full fixture-matrix identity tests, and every
  gate run with directly verified exit codes — format, lint-as-error,
  tests, the full-stream bit-identity replays, coverage and the
  complexity ceiling.
- **Stage R1 — cross-review (Codex).** An independent review of the
  entire package *including the spine*, conducted against the Fortran
  source rather than against the port's own claims: transcription
  fidelity line-by-line, precision-map compliance, state-translation
  compliance, test/evidence alignment. Findings carry severity and
  line evidence; accepted findings are fixed before handback.
- **Stage R2 — final review and close (Claude).** Gates re-run
  independently — reported results are never taken on faith — plus
  targeted source-vs-port reads of the highest-risk translations,
  review of the R1 dispositions, and the close, or a bounce with named
  blockers.

The asymmetry is the point. The two systems have complementary error
profiles — measured in this record, not assumed: the spine author
missed an entire embedded 15-unit numerical library during
decomposition and made the census arithmetic slip, both caught by the
cross-reviewer; the completing executor introduced the two
High-severity transcription errors of §3, caught by the source-reading
review. Each fixed errors the other made. Review has teeth only when
the reviewer is never the author, and the stage boundaries guarantee
that for every line in the repository.

Above the pipeline sits the human layer. No scientific or governance
decision was made autonomously. The operator (Roger Lew, University of
Idaho — also the maintainer of the pinned upstream CLIGEN fixes)
ratified, in the written record: the port posture itself, the fixture
station cohort, the seed semantics, the decision to replace the legacy
command-line interface with a schema-versioned runspec
([SPEC-RUNSPEC](docs/specifications/SPEC-RUNSPEC.md)), every license
adjudication, and every package closure — and dispatched each stage.
Every stage, finding, fix, and gate result is in the work-package
record with command-level evidence.

## 5. The substrate everything above stands on: a persistent, mutable, shared workspace

Nothing in §§1–4 works the way much of agentic-AI practice is
currently organized — ephemeral sessions in which the conversation is
the memory, the human is the only carrier of state between sessions,
and knowledge dies with the context window. This project's substrate
is different, and because it is the part of the methodology its
authors most take for granted, it gets named explicitly:

- **The repository is the working memory.** AI context windows are
  finite and were treated as disposable by design: sessions ended and
  fresh ones began throughout the day, across two vendors' models.
  Every dispatch began "work from the repository record — do not
  assume prior session memory," and that was sufficient, because each
  work package carries its plan, scope, authority citations, evidence
  artifacts, review findings, dispositions, and gate results. The
  record is not documentation *about* the work; it is the medium the
  work happened in.
- **A forward-only queue.** [`docs/ROADMAP.md`](docs/ROADMAP.md) holds
  only what is next; completed items leave it and live in the package
  record. Any agent, human, or auditor gets the project's exact state
  in one read, with no archaeology through stale status.
- **Dispatch as committed artifact.** The inter-agent protocol was
  git, not shared context. Each stage's handoff and kickoff prompt was
  written into the package *after* the code it cites had landed, then
  dispatched by the operator into the other model's fresh session. The
  two AI systems never shared a context window; they coordinated
  exclusively through the record — which is precisely why the §4
  cross-review had teeth (the reviewer's only channel to the author's
  reasoning was the same evidence a future auditor has), and why any
  stage was resumable by any session at any time.
- **The environment learns.** The workspace is mutable in the strong
  sense: process failures became committed policy, in the same medium
  the next session reads, within hours of occurring. A stage executed
  on an unstated branch at one package's review became a standing
  dispatch rule in the package template; the math-library divergence
  became coding-standard §1.3; a lint-vs-fidelity collision became the
  faithful-shape suppression rule. The ratchet turns one way because
  the lessons live where the executors live.

The models mattered (§6). But the substrate is what let capability
compound instead of dissipate: the same models operating in ephemeral
sessions, with a human relaying context by hand, would have produced
fluent volume, no record — and, by §3's own standard, no credibility.

## 6. Why one day was possible

For the historical record, the speed had identifiable causes, none of
which involve skipped verification:

1. **The source was built to be ported.** C. R. Meyer's 1999 recode of
   CLIGEN — executed against written coding conventions — produced
   small, single-purpose, documented units whose structure mapped
   nearly one-to-one onto Rust functions, and whose built-in
   statistical self-tests became verification instruments. See
   [CREDITS.md](CREDITS.md).
2. **Every decision preceded the code that depended on it.** Posture
   before port; fixtures before transcription; gates before functions;
   precision map before arithmetic. The executors never improvised on
   anything load-bearing.
3. **Verification cost collapsed.** Checking 46 million values for
   bit-identity takes 13 seconds in release mode. When re-verification
   is that cheap, aggressive pace stops being risky: every step is
   checked before the next begins.
4. **The moment's tooling.** The port ran on two frontier AI systems
   released to preview within the same week (Anthropic Claude Fable 5;
   OpenAI GPT-5.6-sol), whose complementary error profiles are what
   made the §4 cross-review substantive rather than ceremonial. The
   operator's assessment, after a year of intensive agentic-AI
   development work, is that the numerical-parity chain in §3 was not
   reliably executable by AI systems even a month earlier.

## 7. What is established, and what is not

Established, with re-runnable evidence: the Rust faithful mode
reproduces the pinned CLIGEN 5.32.3 reference build bit-for-bit on the
committed fixture matrix — outputs and instrumented interior state —
on the reference platform (x86-64 Linux, glibc 2.39 runtime as the
behavioral reference; transcendentals pinned in-crate so the Rust side
does not depend on the host's math library).

Not established, stated plainly:

- **Universality over inputs.** The end-to-end guarantee covers the
  fixture matrix. Unit-level guarantees are much broader (tens of
  millions of independent input vectors), but an input regime no
  fixture exercises could, in principle, reach untested behavior.
  Expanding the matrix is mechanical: add a station, regenerate
  goldens, re-run.
- **Cross-platform bit-identity of the Fortran.** The *reference* is
  pinned to one build; other Fortran builds differ (the production
  build's fast-math flags were measured to be output-identical on the
  stochastic fixtures, but that is an observation, not a guarantee).
  The Rust port is designed to be platform-stable where the Fortran is
  not.
- **Scientific adequacy of CLIGEN itself.** This is a fidelity port of
  a fitted stochastic model. It reproduces CLIGEN 5.32.3's behavior,
  including behaviors that are historical accidents (several are
  documented in the work-package record). Whether CLIGEN's
  distributions serve a given application is a question about CLIGEN,
  unchanged by this port. Deliberate improvements are future,
  *labeled* extensions behind versioned generation profiles — never
  silent.

## 8. The code's register: non-idiomatic by design, and a landing point

A Rust programmer opening `crates/cligen/src/` will find distinctly
non-idiomatic code: Fortran-shaped short symbols (`mox`, `iarrct`,
`k7`), `k[1] = 3 * k[1]` where an idiomatic author would write `*=`,
sequential range checks instead of `clamp`, the literal `6.283185`
where the standard library offers `TAU`, and wide `&mut` state-struct
parameter lists threading what were once common blocks. This is
disclosed plainly, with the contention that it is a *register*, not
sloppiness — and the difference is checkable.

The faithful core's intended reader holds `cligen.f` in the other
hand. Source-shape is the correct register for that reader
permanently: it is what makes line-for-line verification — the §2
acceptance regime, and any future auditor's independent read —
tractable. The trade was written down as policy *before the code
existed* (the coding standard's naming and state-translation rules),
and it is enforced per-line: every deviation from Rust idiom carries a
citation to the source line it mirrors, and every lint suppression is
individual, justified, and cited. Unaccountable deviation is what
makes generated code slop; here, "why is this line like this?" always
has an answer of the form `cligen.f:NNNN`.

The faithful core is also a **landing point, not the destination
architecture**. Two already-planned futures draw the boundary. The
extension surfaces — the typed parameter model, the parquet outputs,
the Python API, the runspec CLI — are written for readers who hold no
Fortran, and are held to ordinary idiomatic standards; the codebase is
*supposed* to read as two registers. And when the planned native-f64
mode arrives as an idiomatic, borrow-checker-natural engine, the
faithful core is expected to graduate from product to **oracle**: the
executable specification the new engine is verified against, exactly
the role the Fortran serves for the Rust today. The
extensible-architecture question is deliberately deferred to that
engine rather than forced onto the faithful core — contorting the
verification substrate to be ergonomic would spend its one
non-negotiable property, which is that it looks like its specification.

## 9. Check it yourself

Skeptical review is welcome; the repository is built for it.

```sh
# 1. The committed test surface (fixture samples, direct vectors,
#    the 12-golden end-to-end gate):
cargo test --release

# 2. Rebuild the reference and regenerate the goldens from the vendored
#    public-domain source (provenance and commands:
#    docs/work-packages/20260709-golden-fixture-harness/artifacts/):
gfortran -O0 -ffp-contract=off -fprotect-parens -fno-fast-math \
  -Ireference/cligen532 reference/cligen532/cligen.f -o cligen532-pinned
# ... then run the 12 fixture invocations in fixture-runs.tsv and
# compare hashes against the committed manifest.

# 3. Regenerate the interior tap captures (committed patches applied to
#    a copy of the source; see each package's tap-manifest.md) and run
#    the full-stream identity gates:
cargo test --release -- --ignored --nocapture
```

Beyond re-running gates: read any port module beside its Fortran
(each cites its source lines and carries a symbol glossary — the code
is deliberately shaped for side-by-side reading); read the review
findings and their dispositions; follow a hash from the provenance
records to a file. Anything that does not check out is a bug report we
want.
